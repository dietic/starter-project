# Firestore Schema

## Collections

### `articles` (top-level)

Holds articles written by authenticated users of the app (journalist feature).

| Field            | Type      | Required | Constraints                                                                                                 | Notes                                                                                                                             |
| ---------------- | --------- | -------- | ----------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| `authorId`       | string    | yes      | `== request.auth.uid` on create; **immutable** on update                                                    | Firebase Auth UID of the author. Canonical owner of the article.                                                                  |
| `authorEmail`    | string    | no       | `<= 320 chars`                                                                                              | Snapshot of the author's email at write time. Denormalized so the feed doesn't have to read the auth record.                      |
| `authorName`     | string    | no       | `<= 80 chars`                                                                                               | Snapshot of the author's display name at write time. Preferred in the byline; email is the fallback.                              |
| `authorPhotoUrl` | string    | no       | `startsWith("https://")`                                                                                    | Snapshot of the author's profile photo URL at write time. Optional; the byline renders an initial if absent.                     |
| `title`          | string    | yes      | `1 <= len <= 120`                                                                                           | Plain text.                                                                                                                       |
| `description`    | string    | yes      | `1 <= len <= 500`                                                                                           | Short summary shown in feed tiles.                                                                                                |
| `content`        | string    | yes      | `1 <= len <= 20000`                                                                                         | Article body. Plain text for now; Markdown could be introduced later without a schema change.                                     |
| `thumbnailUrl`   | string    | yes      | `startsWith("https://")`                                                                                    | Public download URL returned by Cloud Storage. Stored here so the feed never needs a Storage SDK call to render thumbnails.       |
| `thumbnailPath`  | string    | yes      | `startsWith("media/articles/")`, **immutable** on update                                                    | The Storage object path. Stored so we can delete the blob when the article is deleted — otherwise the URL orphans its storage.    |
| `publishedAt`    | timestamp | yes      | `== request.time` on create; **immutable** on update                                                        | Server-assigned on create. Used for feed ordering.                                                                                |

**Document ID:** auto-generated by Firestore (20-char random string). Used as the directory fragment inside `media/articles/<docId>/` so an article owns its storage namespace.

### `articles/{articleId}/comments/{commentId}` (subcollection)

| Field       | Type      | Required | Constraints                      | Notes                                                                                                                |
| ----------- | --------- | -------- | -------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| `authorId`        | string    | yes      | `== request.auth.uid` on create  | Author's Firebase Auth UID. Used for delete-own authorization.                                                              |
| `authorName`      | string    | no       | `<= 80 chars`                     | Snapshot of commenter's display name at write time. Shown in the UI byline.                                                |
| `authorPhotoUrl`  | string    | no       | `startsWith("https://")`           | Snapshot of commenter's profile photo URL at write time. Shown as the comment avatar; falls back to an initial if absent. |
| `text`            | string    | yes      | `1 <= len <= 2000`                | The comment body.                                                                                                          |
| `createdAt`       | timestamp | yes      | `== request.time` on create       | Server-assigned on create. Used for chronological ordering.                                                                |
| `replyTo`         | string    | no       | commentId of another comment      | Present only on replies. Enables 1-level threading in the UI (replies are nested under their parent comment).              |
| `isDeleted`       | boolean   | yes      | `false` on create                 | Soft-delete tombstone flag. A deleted parent stays in place so its reply thread remains visible.                          |

**Why a subcollection?** Comments belong to a specific article, so scoping them under the parent doc makes the access pattern natural (`articles/<id>/comments`), keeps rules per-parent, and avoids a top-level collection bloat. Author identity is stored for own-delete authorization; names/avatars are snapshotted for O(1) rendering (same pattern as articles).

**Threading model:** flat, 1-level. A reply has `replyTo: <parentCommentId>`. Rules enforce that `replyTo` points to an existing top-level comment on the same article. Only the article author may create replies; other authenticated users may only create top-level comments.

**Deletion model:** comments are soft-deleted only. Deleting a comment turns it into a tombstone (`isDeleted: true`, empty `text`) instead of removing the document. This keeps threaded conversations stable and prevents orphaned replies.

## Cloud Storage Layout

```
media/
  articles/
    <uid>/
      <articleId>/
        thumbnail.<ext>
  users/
    <uid>/
      avatar.<ext>
```

- One storage path per article nested under the author's UID. Storage rules use that UID segment to enforce owner-only writes/deletes before the Firestore article document exists.
- Public-read: the thumbnail URL is embedded in the Firestore doc and rendered in every feed tile. Making it public avoids signed-URL churn.
- Authenticated-write only: rules enforce `request.auth != null`.
- Content-type limited to `image/*`; size capped at 5 MB.

## Design notes

- **Why top-level `articles` instead of `users/{uid}/articles`?** The primary read pattern is "give me all articles for the feed," not "give me my articles." A top-level collection serves that with a single query. Author filtering (`my articles`) is a `where('authorId', '==', uid)` query.
- **Why snapshot `authorEmail`?** The alternative is reading `users/{uid}` on every feed render — N+1 reads. Snapshotting trades minor staleness (if a user changes email, old articles still show the old one) for O(1) reads. Acceptable given how rarely emails change.
- **Why store `thumbnailPath` alongside `thumbnailUrl`?** URLs contain Firebase tokens that can be rotated; paths are stable. Deletion uses the path; rendering uses the URL. One canonical owner per concern.
- **Immutable fields on update.** `authorId`, `publishedAt`, and `thumbnailPath` can never change after create. Enforced in `firestore.rules`. Prevents a compromised client from reassigning ownership or backdating articles.
- **Comments use soft-delete only when needed.** This keeps threaded conversations stable without giving clients arbitrary comment edit powers.

## Indexes

None required beyond Firestore's automatic single-field indexes. The two queries we run:

1. Feed: `orderBy('publishedAt', descending: true)` — single-field index, auto-created.
2. My articles: `where('authorId', '==', <uid>).orderBy('publishedAt', descending: true)` — requires a composite index on `(authorId, publishedAt desc)`. Firestore will prompt to create it on first run; index definition will be added to `firestore.indexes.json` at deploy time.
